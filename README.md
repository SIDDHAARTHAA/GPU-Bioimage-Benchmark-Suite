# GPU Bioimage Benchmark Project

This project is a demo-first, benchmark-first build around one idea:

- the same 3D bioimage workflow should run on CPU or GPU
- the wrapper layer should stay close to raw backend performance
- the wrapper should be easier to use than writing separate CPU and GPU code

## What We Are Proving

There are two different benchmark stories in this repo.

### 1. Four-way benchmark

This proves wrapper overhead is small.

For each operation we time:

- raw CPU
- `cubic` CPU
- raw GPU
- `cubic` GPU

If `cubic CPU` is close to raw CPU, and `cubic GPU` is close to raw GPU, then the wrapper is thin.

Where to read the code behind those four columns:

- benchmark harness:
  - `scripts/benchmark_four_way_backend.py`
- raw CPU paths:
  - `skimage.transform.rescale`
  - `scipy.ndimage.gaussian_filter`
  - `scipy.ndimage.median_filter`
- raw GPU paths:
  - `cucim.skimage.transform.rescale`
  - `cupyx.scipy.ndimage.gaussian_filter`
  - `cupyx.scipy.ndimage.median_filter`
- `cubic` wrapper paths:
  - `vendor/cubic/cubic/skimage.py`
  - `vendor/cubic/cubic/scipy.py`
  - `vendor/cubic/cubic/cuda.py`

Plain meaning:

- raw CPU calls the standard CPU library directly
- raw GPU calls the CUDA-backed GPU library directly
- `cubic` calls one wrapper function name
- the wrapper checks whether the array lives on CPU or GPU, then routes to the matching backend

Current operations:

- `rescale_down_order3`
- `rescale_up_order3`
- `gaussian_filter_sigma_1.2`
- `median_filter_size5`

Latest observed wrapper-overhead range on this machine:

- CPU overhead: about `-3.80%` to `+0.37%`
- GPU overhead: about `-5.90%` to `+0.41%`

Interpretation:

- the wrapper cost is basically negligible here
- negative overhead just means tiny timing noise / caching effects, not “free speed”

### 2. Rescale speedup benchmark

This proves GPU acceleration is large for a real 3D image operation.

This benchmark runs only:

- `cubic CPU`
- `cubic GPU`

for 3D rescaling across interpolation orders `0..5`.

It answers:

- how much faster is GPU than CPU for 3D rescaling?
- how does that change with interpolation order?
- how does that change with image size?

Latest observed `CPU / GPU` speedups on this machine:

- overall: about `18.16x` to `106.09x`
- upscale: about `18.16x` to `52.11x`
- downscale: about `23.64x` to `106.09x`

### Why Python 3.13 is used

The exact blocker on system Python `3.14` was:

- `cucim-cu13`

That package did not work cleanly in the `3.14` setup here, so the safe fix was:

- keep system Python untouched
- use a local Python `3.13`
- create `.venv313` from it

Additional CUDA runtime pieces were also needed for the GPU stack to fully work, including FFT-backed deconvolution.

## What Each Process Does To The Image

This is the non-scientist explanation.

### `rescale_down_order3`

Meaning:

- shrink the image

What it does visually:

- image gets smaller
- fine detail is reduced
- often looks blurrier because some information is removed

So yes, this should usually look less sharp than the original.

### `rescale_up_order3`

Meaning:

- enlarge the image

What it does visually:

- fills in extra pixels by interpolation
- can look smoother or softer
- does not create true new detail

So it is normal if the enlarged image looks “cleaner but softer”.

### `gaussian_filter_sigma_1.2`

Meaning:

- blur the image in a controlled way

What it does visually:

- reduces high-frequency noise
- smooths edges and textures
- makes the image look softer

This is expected to look blurrier than the original.

### `median_filter_size5`

Meaning:

- replace each pixel using the median value in a neighborhood

What it does visually:

- removes isolated noisy speckles
- can preserve edges better than Gaussian blur
- still changes local texture and can look a bit flatter or cleaner

### `richardson_lucy_iter10`

Meaning:

- iterative deblurring

What it tries to do:

- reverse blur using a blur model called a `PSF`

What it does visually:

- can make structures look sharper
- can also create ringing or weird artifacts if the PSF is not a good match

That is why our current deconvolution may not look perfectly “better”:

- the current PSF is synthetic, not the true microscope PSF for that image

### Segmentation

Meaning:

- turn an intensity image into labeled objects

For nuclei segmentation:

- use the DNA channel
- threshold bright nucleus regions
- separate touching nuclei
- clean up tiny mistakes

For cell segmentation:

- use membrane + DNA + mitochondria information
- build a mask of likely cells
- use nuclei as seeds
- expand them into whole-cell regions

The output is not a prettier image.
It is a labeled map saying “this voxel belongs to object 1, object 2, object 3, ...”.

### 2. Run the four-way benchmark

```bash
./scripts/run_four_way_benchmark.sh
```

What it does:

- loads the real `cells3d` benchmark volume
- creates CPU and GPU copies of the same data
- times raw CPU vs `cubic` CPU vs raw GPU vs `cubic` GPU
- prints live progress logs while it runs
- stages benchmark artifacts outside `results/` while the timings are still running
- moves the JSON, CSV, and PNG files into `results/` only after the benchmark fully completes

Files written:

- `results/four_way_benchmark.csv`
- `results/four_way_benchmark.json`
- `results/four_way_benchmark_meta.json`
- `results/four_way_rescale_down_order3.png`
- `results/four_way_rescale_up_order3.png`
- `results/four_way_gaussian_filter_sigma_1.2.png`
- `results/four_way_median_filter_size5.png`

Use the JSON if you want a cleaner machine-readable summary than CSV.

Timing note:

- `four_way_benchmark.json` stores per-operation timings (median and measured totals)
- `four_way_benchmark_meta.json` stores full wall-clock runtime for the whole run
- wall-clock is always higher than sum of medians because the full run includes:
  - warmups and repeated calls
  - dataset prep and CPU/GPU transfers
  - plotting and file writes
  - Python/import/runtime overhead

### 3. Run the cubic CPU vs cubic GPU rescale benchmark

```bash
./scripts/run_rescale_benchmark.sh
```

What it does:

- uses the same real `cells3d` volume
- runs only `cubic CPU` and `cubic GPU`
- tries interpolation orders `0..5`
- prints live timing logs while running

Important:

- this benchmark is not raw-vs-wrapper
- this one is only `cubic CPU` vs `cubic GPU`
- the JSON includes `cpu_backend` and `gpu_backend` fields to make that explicit

Files written:

- `results/rescale_benchmark.json`
- `results/rescale_benchmark.png`

## Utilization Note

Seeing CPU/GPU utilization below 100% is normal for these workflows.

Why:

- many operations are memory-bandwidth-bound, not pure compute-bound
- kernels are short and launched sequentially from Python
- benchmark synchronization for accurate timing adds idle gaps
- some phases are CPU-heavy (I/O, plotting, preprocessing), so GPU waits

## Under The Hood

The wrapper idea is simple:

- if the input array is on CPU, call CPU libraries
- if the input array is on GPU, call GPU libraries

That is what makes `cubic` useful:

- you keep one workflow style
- you get CPU or GPU execution depending on where the array lives
- the benchmarks here show that the wrapper overhead stays very small
